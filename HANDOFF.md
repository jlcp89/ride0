### Decisions
- **Status semantic model is now explicit and authoritative** — `to-pickup` = ride booked, driver accepted, rider NOT in car; `en-route` = trip in progress, rider in car; `dropoff` = trip completed. This is documented in the seed_db.py module docstring. Previously the three statuses existed as a VARCHAR enum with no defined meaning (requirement.md:39 listed them as examples with `e.g.` and no semantics). The new model anchors each status to a real trip phase and drives both the seed and the UI.
- **Renamed `pickup` → `to-pickup` with a hyphen, not a space** — user said "to pickup" but a hyphen avoids URL encoding (`?status=to%20pickup` ugly), matches the existing `en-route` convention, and is easier to grep/assert. If the user ever prefers the space version it's a trivial sed across the same files.
- **5-event canonical lifecycle**: `Ride requested` → `Ride accepted by driver` → `Status changed to pickup` → `Status changed to dropoff` → `Ride rated by rider`. Each ride gets only the events that have "happened by now" given its current status (to-pickup = 2 events, en-route = 3, dropoff = 5). This is enforced via a `_create_events_for_ride(ride, duration, now)` helper that knows the full lifecycle in one place.
- **The event descriptions `"Status changed to pickup"` / `"Status changed to dropoff"` are NOT renamed** — they're spec-fixed by requirement.md line 104 and the bonus SQL report (`bonus_report.sql`) filters on the exact strings. The fact that they're now semantically stale under the new status name (the ride doesn't literally change TO `pickup` anymore, it changes FROM `to-pickup` TO `en-route` at the pickup moment) is a cost we pay for spec compatibility. A comment in the seed_db docstring flags this.
- **Wipe-and-rebuild idempotency, not skip-if-exists** — `User.objects.all().delete()` at the top of `seed_db.handle()` cascades to rides and events. This fixes the session-5 footgun where `seed_db` no-opped on re-runs because the old `if User.objects.exists(): return` guard. Now every deploy produces fresh timestamps. Safe because this seed command is only ever used for the demo DB. If the project ever productionizes, remove seed_db from deploy.yaml first.
- **Reset AUTO_INCREMENT after delete via vendor-specific raw SQL** — SQLite doesn't reset `sqlite_sequence` on DELETE and MySQL doesn't reset `AUTO_INCREMENT` on DELETE. Without this, re-seeds accumulated to rides 121-144 in testing. Fix: `_reset_auto_increment()` helper detects `connection.vendor` and issues `DELETE FROM sqlite_sequence WHERE name IN (...)` or `ALTER TABLE ... AUTO_INCREMENT = 1`. Django has no built-in for this. Verified by running seed_db twice and confirming IDs land at 1-24.
- **Rider events dropped from 81 to 81 but total reduced from the Phase 1 interim of 45 → 81 is correct** — wait, the journey was: original seed = 68 events, Phase 1 interim = 45 events, Phase 2 final = 81 events. 81 is larger because every ride now has at least 2 events (request + accept) where en-route and to-pickup rides previously had 0-1. More events but every event is semantically meaningful, not stress-test bloat.
- **Ride 1 no longer has 18 events** — Phase 1 left ride 1 at 18 events (old "stress test" carryover); user pushed back ("does this have sense?") because 18 events on a single ride is unrealistic for a demo. Phase 2 reduced ride 1 to the standard 5-event lifecycle via the uniform helper. Stress-testing the Prefetch filter is now the responsibility of `tests/test_performance.py::test_prefetch_filters_created_at`, not the seed data.
- **One atomic commit per repo, not split into 9 steps** — the session 6 HANDOFF recommended splitting the ride1 UX overhaul into 9 commits, but the user already committed session 6's work before session 7 started (ride1 git status at session start showed only the 5 files I renamed, not ~20). For session 7, the changes form a single logical unit — splitting leaves broken intermediate states (test_views.py asserts against `to-pickup` but seed_db.py still creates `pickup`, etc.). Commit messages drafted: `seed: rename pickup to to-pickup, add 5-event lifecycle` (ride0) and `ui: rename pickup status to to-pickup` (ride1).
- **Left `.claude/settings.local.json` alone** — the permission allowlist has a stale `curl 'http://107.23.122.99/api/rides/?status=pickup'` entry. Silently modifying permission configs without explicit user consent is risky, even for a cosmetic string change. User can update manually if desired.
- **Left `requirement.md:39` alone** — it's the original take-home spec document ("e.g., 'en-route', 'pickup', 'dropoff'"). Rewriting the spec to match my rename would be dishonest and confuses provenance.
